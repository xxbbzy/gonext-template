# JWT Module Unit Test Design

Date: 2026-03-28
Scope: `backend/pkg/jwt`
Status: Draft for implementation planning

## 1. Background And Goal

Current `backend/pkg/jwt` provides three core capabilities:

- Generate access token (`GenerateAccessToken`)
- Generate refresh token (`GenerateRefreshToken`)
- Parse token (`ParseToken`)

The goal is to add unit tests that cover core token flow behavior with a black-box-first strategy, ensuring token issuance and parsing behavior is verifiable and stable.

## 2. In Scope / Out Of Scope

In scope:

- Access token generation success path
- Refresh token generation success path
- Valid token parsing success path
- Expired token behavior (parse success before expiry, then parse failure after expiry)
- Invalid token behavior:
  - malformed token string
  - signature mismatch token
- Optional lightweight constructor validation:
  - invalid access duration
  - invalid refresh duration

Out of scope:

- Refactoring JWT production code
- Changing claim schema or signing algorithm
- Integration tests across middleware/handler layers

## 3. Recommended Approach

Use a black-box-first table-driven test design in a new file:

- `backend/pkg/jwt/jwt_test.go`

Why this approach:

- Aligned with repository test style (stdlib `testing`, focused assertions)
- Resilient to internal refactors because tests depend on public API contracts
- Easy to expand with additional token scenarios later

## 4. Test Architecture

### 4.1 Test Grouping

Define two main test groups:

1. `TestManager_GenerateTokens`
2. `TestManager_ParseToken`

### 4.2 Shared Test Helpers

Introduce small local helpers to keep cases readable:

- `newTestManager(t, accessTTL, refreshTTL) *Manager`
- `mustGenerateAccessToken(t, m, userID, role) string`
- `mustGenerateRefreshToken(t, m, userID, role) string`

Helpers are package-local in the test file only.

## 5. Case Matrix And Assertions

## 5.1 `TestManager_GenerateTokens`

Case A: generate access token

- Setup: manager with `access=15m`, `refresh=24h`
- Action: call `GenerateAccessToken(42, "admin")`
- Assert:
  - no error
  - token string not empty
  - parse token with same manager succeeds
  - parsed `UserID == 42`, `Role == "admin"`

Case B: generate refresh token

- Setup: same manager
- Action: call `GenerateRefreshToken(42, "admin")`
- Assert:
  - no error
  - token string not empty
  - parse token with same manager succeeds
  - parsed `UserID == 42`, `Role == "admin"`

## 5.2 `TestManager_ParseToken`

Case A: parse valid token

- Setup: manager + generated access token
- Action: call `ParseToken(token)`
- Assert:
  - no error
  - claims are non-nil and match source identity

Case B: expired token after ttl

- Setup: manager with very short access TTL (example `20ms`)
- Action:
  - generate access token
  - immediately parse once (must succeed)
  - sleep beyond TTL buffer (example `40ms`)
  - parse again
- Assert:
  - second parse returns error
  - error message contains `token expired`

Case C: malformed token

- Setup: manager
- Action: `ParseToken("not-a-jwt")`
- Assert:
  - returns error
  - error message contains `invalid token`

Case D: invalid signature token

- Setup:
  - manager A with `secret-a`
  - manager B with `secret-b`
  - token generated by manager A
- Action: parse token using manager B
- Assert:
  - returns error
  - error message contains `invalid token`

## 5.3 Optional Constructor Validation

`TestNewManager_InvalidDuration`:

- invalid access duration should return error
- invalid refresh duration should return error

This is optional but recommended due to low cost and high signal.

## 6. Data Flow And Error Handling Expectations

- Generation tests validate output token usability only through public parse API.
- Parse tests verify behavioral contract, not JWT internal binary layout.
- Error assertions should focus on stable error intent (`token expired`, `invalid token`) instead of third-party library internals.

## 7. Risks And Mitigations

Risk: flaky expiration timing under slow CI.

Mitigations:

- Use short but not extreme TTL values (for example 20ms + 40ms wait)
- Keep one immediate parse before sleep to prove token is valid pre-expiry
- Avoid overly strict wall-clock boundary assertions
